Focus Mode — Alternatywy bez hover overlay

Problem: hover overlay na barze koliduje z drag-to-move. Najeżdżasz żeby przesunąć okno, a przyciski się pojawiają i przechwytują kliknięcia. Poniżej 3 podejścia bez tego problemu.

Problem — obecny hover overlay (reference)

Hover na barze = overlay zasłania treść + koliduje z drag area. Przyciski migają, nie da się stabilnie przesunąć okna.

Opcja A — Przyciski zawsze widoczne + dropdown na linki

✓ i ✕ zawsze widoczne po prawej stronie (za time pill).
Kliknięcie w kod projektu (np. "TOP") otwiera dropdown z linkami (VS Code, iTerm, Obsidian note).
Cały bar poza przyciskami = drag area. Zero hover overlay.

Opcja A — zalety:
+ Przyciski ✓ ✕ od razu widoczne — nie trzeba hovera
+ Cały bar poza 2 przyciskami = drag area
+ Linki ukryte za kliknięciem w kod projektu (dropdown, nie overlay)
+ Najprostsze w implementacji

Wady:
− Bar trochę szerszy (2 przyciski zajmują ~60px)
− Dropdown wymaga kliknięcia (dodatkowy krok dla linków)

Opcja B — Prawy klik = context menu

Cały bar to czysta drag area. Zero przycisków widocznych na stałe.
Prawy klik → natywne-style context menu z wszystkimi akcjami i linkami.
Najczystszy bar, ale mniej discoverable.

Opcja B — zalety:
+ Najczystszy bar — zero clutter, max miejsca na tytuł
+ 100% powierzchni = drag area
+ Shortcuty (⌘D, Esc) przy pozycjach menu — uczy skrótów
+ Można dodać menu nativeowe (Electron Menu) zamiast custom

Wady:
− Prawy klik nie jest oczywisty (discoverability)
− Nie widać przycisku "done" — trzeba wiedzieć o prawy klik
− Na trackpadzie prawy klik jest mniej naturalny

Opcja C — Hybrid: przyciski + "more" dropdown

✓ Done i ✕ Exit zawsze widoczne z labelem. Przycisk "•••" otwiera dropdown z linkami projektu.
Trzy interaktywne elementy po prawej, reszta = drag. Jasna hierarchia: akcje > linki.

Opcja C — zalety:
+ Przyciski z labelem — czytelne, nie trzeba hovera
+ "•••" to znany wzorzec dla "more actions"
+ Czysta hierarchia: główne akcje widoczne, linki za 1 kliknięciem

Wady:
− 3 przyciski po prawej to dużo — bar jest gęstszy
− "•••" + ✓ + ✕ = 3 klikalne elementy obok siebie (łatwo chybić)
Rekomendacja: Opcja A

Najprostsze i najstabilniejsze rozwiązanie:
− ✓ ✕ zawsze widoczne — zero zaskoczenia, zero migania
− Kliknięcie w kod projektu → dropdown z linkami (1 krok)
− Cały bar poza przyciskami = pewny drag
− Łatwe do zaimplementowania, zero edge case'ów z hover state

Opcja B (context menu) jest najczystsza wizualnie, ale mniej discoverable.
